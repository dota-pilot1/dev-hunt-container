Level 1

프론트 로직 vs 백엔드 로직

Level 1의 목표는 프론트가 STOMP로 서버에 연결하고, 서버가 받은 ping 메시지를 pong 메시지로 다시 내려주는 것이다. 프론트 핵심은 connect()이고, 백엔드 핵심은 WebSocketConfig와 ChatStudyMessageController다.

한 줄 요약

프론트는 연결하고, 구독하고, 메시지를 보낸다. 백엔드는 연결 endpoint를 열고, 메시지 라우팅 규칙을 만들고, 들어온 메시지를 응답 topic으로 보낸다.

프론트와 백엔드 역할 비교

프론트 핵심 역할

  • @stomp/stompjs의 Client 생성
  • ws://localhost:4101/ws로 연결 시작
  • 연결 성공 후 /topic/chat.level1.pong 구독
  • Ping 버튼 클릭 시 /app/chat.level1.ping으로 메시지 발행
  • 받은 pong 메시지를 화면 로그에 출력

백엔드 핵심 역할

  • /ws WebSocket endpoint 오픈
  • /app으로 들어오는 메시지를 Controller로 라우팅
  • /topic 구독자에게 메시지를 브로드캐스트
  • @MessageMapping으로 ping 메시지 처리
  • @SendTo로 pong 메시지 전송

프론트 핵심 코드

connect 함수

const connect = () => {
  if (clientRef.current?.active) return;

  setConnectionState("connecting");

  const client = new Client({
    brokerURL: wsUrl,
    reconnectDelay: 0,
    debug: (line) => {
      if (process.env.NODE_ENV === "development") {
        console.debug("[stomp]", line);
      }
    },
    onConnect: () => {
      setConnectionState("connected");
      appendLog("system", "STOMP 연결 완료");
      client.subscribe("/topic/chat.level1.pong", (frame: IMessage) => {
        const body = JSON.parse(frame.body) as PongMessage;
        appendLog(
          "received",
          `${body.sender}: ${body.message} (${new Date(body.serverTime).toLocaleTimeString("ko-KR")})`
        );
      });
    },
    onDisconnect: () => {
      setConnectionState("disconnected");
      appendLog("system", "STOMP 연결 해제");
    },
    onStompError: (frame) => {
      setConnectionState("error");
      appendLog("error", frame.headers.message ?? "STOMP 프로토콜 에러");
    },
    onWebSocketError: () => {
      setConnectionState("error");
      appendLog("error", "WebSocket 연결 에러");
    },
    onWebSocketClose: () => {
      if (connectionState !== "idle") {
        setConnectionState("disconnected");
      }
    },
  });

  clientRef.current = client;
  client.activate();
};

이 함수가 하는 일을 순서대로 보면 다음과 같다.

  1. clientRef.current?.active로 중복 연결을 막는다.
  2. 화면 상태를 connecting으로 바꾼다.
  3. new Client(...)로 STOMP client 객체를 만든다.
  4. onConnect 안에서 연결 성공 후 topic을 구독한다.
  5. clientRef.current = client로 연결 객체를 저장한다.
  6. client.activate()로 실제 연결을 시작한다.

프론트의 publish 코드

client.publish({
  destination: "/app/chat.level1.ping",
  body: JSON.stringify({
    sender,
    message,
  }),
});

이 코드는 서버의 @MessageMapping("/chat.level1.ping")으로 메시지를 보낸다. 프론트에서는 /app prefix를 붙이고, 백엔드 Controller에서는 prefix를 뺀 주소로 매핑한다.

백엔드 핵심 코드

WebSocketConfig

@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("http://localhost:*", "http://127.0.0.1:*");
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.setApplicationDestinationPrefixes("/app");
        registry.enableSimpleBroker("/topic", "/queue");
        registry.setUserDestinationPrefix("/user");
    }
}

이 설정의 의미는 다음과 같다.

설정 의미 프론트와 매칭
addEndpoint("/ws") WebSocket 접속 endpoint를 연다. brokerURL: "ws://localhost:4101/ws"
setApplicationDestinationPrefixes("/app") 클라이언트가 서버 Controller로 메시지를 보낼 때 쓰는 prefix다. destination: "/app/chat.level1.ping"
enableSimpleBroker("/topic", "/queue") 서버가 구독자에게 메시지를 뿌릴 수 있는 broker prefix다. subscribe("/topic/chat.level1.pong")
setUserDestinationPrefix("/user") 나중에 사용자별 1:1 메시지로 확장할 때 쓰는 prefix다. Level 1에서는 아직 사용하지 않음

ChatStudyMessageController

@Controller
public class ChatStudyMessageController {

    @MessageMapping("/chat.level1.ping")
    @SendTo("/topic/chat.level1.pong")
    public ChatLevel1PongResponse ping(ChatLevel1PingRequest request) {
        String sender = request.sender() == null || request.sender().isBlank()
                ? "anonymous"
                : request.sender().trim();
        String message = request.message() == null || request.message().isBlank()
                ? "ping"
                : request.message().trim();

        return new ChatLevel1PongResponse(sender, "pong: " + message, Instant.now());
    }
}

이 컨트롤러가 하는 일을 순서대로 보면 다음과 같다.

  1. 프론트가 /app/chat.level1.ping으로 메시지를 보낸다.
  2. /app prefix는 Spring 메시지 broker 설정에서 제거된다.
  3. 남은 /chat.level1.ping이 @MessageMapping과 매칭된다.
  4. JSON body가 ChatLevel1PingRequest로 변환된다.
  5. 서버가 ChatLevel1PongResponse를 만든다.
  6. @SendTo에 의해 /topic/chat.level1.pong 구독자에게 응답이 전송된다.

주소 매칭표

목적 프론트 백엔드 설명
WebSocket 연결 ws://localhost:4101/ws addEndpoint("/ws") 처음 연결하는 endpoint
서버로 메시지 보내기 /app/chat.level1.ping @MessageMapping("/chat.level1.ping") /app은 서버 Controller로 보내는 prefix
서버 응답 받기 subscribe("/topic/chat.level1.pong") @SendTo("/topic/chat.level1.pong") 서버가 topic 구독자에게 응답 브로드캐스트

구독 요청 처리 집중 설명

구독 요청은 어디서 보내나?

구독 요청은 프론트의 onConnect 안에서 보낸다. 연결이 성공한 뒤 아래 코드가 실행되면 STOMP client는 서버로 SUBSCRIBE 프레임을 보낸다. 

onConnect: () => {
  setConnectionState("connected");
  appendLog("system", "STOMP 연결 완료");

  client.subscribe("/topic/chat.level1.pong", (frame: IMessage) => {
    const body = JSON.parse(frame.body) as PongMessage;
    appendLog(
      "received",
      `${body.sender}: ${body.message} (${new Date(body.serverTime).toLocaleTimeString("ko-KR")})`
    );
  });
}

여기서 핵심은 client.subscribe("/topic/chat.level1.pong", callback)이다. 이 코드는 “나는 앞으로 /topic/chat.level1.pong으로 오는 메시지를 받겠다”라고 서버에 등록하는 요청이다.

구독 요청을 백엔드 Controller가 받는가?

아니다. 이 부분이 헷갈리기 쉽다. 구독 요청은 @MessageMapping 컨트롤러가 직접 받는 것이 아니다.

프론트가 /topic/chat.level1.pong을 구독하면, Spring의 STOMP 메시지 브로커가 해당 WebSocket 세션을 topic 구독자로 등록한다. 현재 프로젝트에서는 이 설정이 아래 코드다. 

registry.enableSimpleBroker("/topic", "/queue");

즉 /topic으로 시작하는 주소는 서버 Controller 메서드로 들어가는 요청이 아니라, Spring simple broker가 관리하는 구독 목적지다.

@MessageMapping은 클라이언트가 /app/...으로 메시지를 보낼 때 실행된다. /topic/... 구독 등록은 simple broker가 처리한다.

구독 요청의 실제 의미

프론트 코드:

client.subscribe("/topic/chat.level1.pong", callback)

이 코드는 내부적으로 이런 의미를 가진 STOMP 프레임을 서버로 보낸다.

SUBSCRIBE
destination:/topic/chat.level1.pong
id:sub-0

서버는 이 프레임을 받고 현재 WebSocket 세션을 /topic/chat.level1.pong 목적지의 구독자 목록에 등록한다.

이후 서버가 같은 destination으로 메시지를 보내면, simple broker는 그 topic을 구독 중인 모든 WebSocket 세션에게 메시지를 전달한다.

그럼 서버는 언제 구독자에게 메시지를 보내나?

사용자가 Ping 버튼을 누르면 프론트가 서버 Controller로 메시지를 보낸다. 

client.publish({
  destination: "/app/chat.level1.ping",
  body: JSON.stringify({
    sender,
    message,
  }),
});

이 메시지는 /app prefix 때문에 백엔드의 @MessageMapping으로 라우팅된다. 

@MessageMapping("/chat.level1.ping")
@SendTo("/topic/chat.level1.pong")
public ChatLevel1PongResponse ping(ChatLevel1PingRequest request) {
    return new ChatLevel1PongResponse(sender, "pong: " + message, Instant.now());
}

여기서 @SendTo("/topic/chat.level1.pong")이 중요하다. 이 메서드의 반환값은 /topic/chat.level1.pong으로 전송된다.

그리고 이 topic은 프론트가 이미 구독해 둔 주소다. 그래서 프론트의 subscribe callback이 실행된다.

구독 처리 흐름을 순서로 보면

  1. 프론트가 client.activate()로 STOMP 연결을 시작한다.
  2. 연결 성공 후 onConnect가 실행된다.
  3. 프론트가 client.subscribe("/topic/chat.level1.pong", callback)을 실행한다.
  4. 브라우저가 서버로 STOMP SUBSCRIBE 프레임을 보낸다.
  5. Spring simple broker가 현재 WebSocket 세션을 /topic/chat.level1.pong 구독자로 등록한다.
  6. 이 시점에는 아직 Controller 메서드는 실행되지 않는다.
  7. 사용자가 Ping 버튼을 누르면 프론트가 /app/chat.level1.ping으로 메시지를 보낸다.
  8. 그제서야 @MessageMapping("/chat.level1.ping") 메서드가 실행된다.
  9. 메서드 반환값이 @SendTo("/topic/chat.level1.pong")으로 전송된다.
  10. simple broker가 이 topic을 구독 중인 프론트 세션에 메시지를 전달한다.
  11. 프론트의 subscribe callback이 실행된다.

구독과 메시지 발행의 차이

구분 프론트 코드 서버 처리 주체 Controller 실행 여부
구독 client.subscribe("/topic/chat.level1.pong", callback) Spring simple broker 실행 안 됨
발행 client.publish({ destination: "/app/chat.level1.ping" }) @MessageMapping Controller 실행됨
응답 전송 프론트가 직접 하지 않음 @SendTo + simple broker Controller 반환 후 broker가 전달

구독 요청을 이해할 때 가장 중요한 문장

subscribe("/topic/...")는 서버 Controller를 호출하는 코드가 아니라, 서버 broker에게 “이 topic의 메시지를 나한테 보내줘”라고 등록하는 코드다.

실제 서버 로직 호출은 publish("/app/...")가 한다. 그리고 Controller가 @SendTo("/topic/...")으로 응답하면, 그때 broker가 구독자들에게 메시지를 뿌린다.

실제 메시지 이동 과정

  1. 프론트가 client.activate()로 /ws에 연결한다.
  2. 연결 성공 시 프론트 onConnect가 실행된다.
  3. 프론트가 /topic/chat.level1.pong을 구독한다.
  4. 사용자가 Ping 버튼을 누른다.
  5. 프론트가 /app/chat.level1.ping으로 JSON 메시지를 보낸다.
  6. 백엔드가 @MessageMapping("/chat.level1.ping")에서 메시지를 받는다.
  7. 백엔드가 ChatLevel1PongResponse를 반환한다.
  8. Spring이 반환값을 /topic/chat.level1.pong으로 전송한다.
  9. 프론트의 subscribe callback이 실행된다.
  10. 프론트가 받은 메시지를 로그에 출력한다.

DTO 매칭

프론트에서 보내는 payload

{
  "sender": "tester",
  "message": "hello websocket stomp"
}

백엔드 요청 DTO

public record ChatLevel1PingRequest(
        String sender,
        String message
) {
}

백엔드 응답 DTO

public record ChatLevel1PongResponse(
        String sender,
        String message,
        Instant serverTime
) {
}

프론트에서 받는 타입

type PongMessage = {
  sender: string;
  message: string;
  serverTime: string;
};

중요한 관점

  • 프론트의 connect()는 서버에 붙고, topic을 구독하는 로직이다.
  • 백엔드의 WebSocketConfig는 연결 주소와 메시지 라우팅 규칙을 여는 로직이다.
  • 백엔드의 ChatStudyMessageController는 실제 ping 메시지를 받아 pong으로 응답하는 로직이다.
  • /app은 클라이언트가 서버 메서드로 보낼 때 붙는 prefix다.
  • /topic은 서버가 구독자에게 내려줄 때 쓰는 prefix다.

서버 로직이 어디냐는 질문에 대한 답

Level 1의 서버 로직은 크게 두 곳이다.

  1. WebSocketConfig.java
    /ws, /app, /topic 같은 STOMP 서버 규칙을 설정한다.
  2. ChatStudyMessageController.java
    프론트가 보낸 /app/chat.level1.ping 메시지를 받아 /topic/chat.level1.pong으로 응답한다.
REST API처럼 @GetMapping, @PostMapping으로 찾으면 안 된다. STOMP 메시지는 @MessageMapping으로 받는다.
