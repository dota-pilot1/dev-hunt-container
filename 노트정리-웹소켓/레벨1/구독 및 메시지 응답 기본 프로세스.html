Message Flow

구독 및 메시지 응답 기본 프로세스

이 문서는 Level 1의 가장 중요한 메시지 흐름을 정리한다. 프론트가 publish로 메시지를 보내고, 백엔드가 @MessageMapping으로 받은 뒤, @SendTo로 topic에 응답하고, 프론트 구독자가 받는 흐름이다.

전체 흐름

1. 프론트가 메시지 보냄 client.publish({ destination: "/app/chat.level1.ping" }) │ ▼ 2. 백엔드가 받음 @MessageMapping("/chat.level1.ping") │ ▼ 3. 백엔드 메서드 실행 public ChatLevel1PongResponse ping(...) │ ▼ 4. 반환값 생성 return new ChatLevel1PongResponse(...) │ ▼ 5. 반환값을 topic으로 전송 @SendTo("/topic/chat.level1.pong") │ ▼ 6. 프론트 구독자가 받음 client.subscribe("/topic/chat.level1.pong", callback)

이 프로세스의 이름

이 흐름은 STOMP 기반 메시지 요청-응답 프로세스라고 보면 된다.

HTTP REST처럼 요청하면 같은 HTTP 응답으로 바로 받는 구조가 아니다. 프론트는 /app/...으로 메시지를 보내고, 서버는 /topic/...으로 메시지를 다시 발행한다. 프론트는 미리 구독해둔 callback으로 그 응답을 받는다.

핵심은 “보내는 주소”와 “받는 주소”가 다르다는 것이다. 보내기는 /app/chat.level1.ping, 받기는 /topic/chat.level1.pong이다.

구분 파일 역할

1. 프론트가 구독을 먼저 등록

메시지 응답을 받으려면 프론트가 먼저 응답 topic을 구독해야 한다. 이 코드는 연결 성공 후 onConnect 안에서 실행된다. 

client.subscribe("/topic/chat.level1.pong", (frame: IMessage) => {
  const body = JSON.parse(frame.body) as PongMessage;
  appendLog(
    "received",
    `${body.sender}: ${body.message} (${new Date(body.serverTime).toLocaleTimeString("ko-KR")})`
  );
});

이 코드는 “/topic/chat.level1.pong으로 오는 메시지를 받겠다”는 등록이다. 서버 Controller를 직접 호출하지 않는다.

2. 프론트가 메시지를 보냄

사용자가 Ping 버튼을 누르면 프론트가 서버로 메시지를 보낸다. 

client.publish({
  destination: "/app/chat.level1.ping",
  body: JSON.stringify({
    sender,
    message,
  }),
});

여기서 destination은 /app/chat.level1.ping이다. /app은 백엔드 설정에서 서버 애플리케이션 메서드로 라우팅하라고 정한 prefix다. 

registry.setApplicationDestinationPrefixes("/app");

3. 백엔드가 메시지를 받음

프론트가 /app/chat.level1.ping으로 보낸 메시지는 백엔드에서 @MessageMapping("/chat.level1.ping")과 매칭된다. 

@MessageMapping("/chat.level1.ping")
public ChatLevel1PongResponse ping(ChatLevel1PingRequest request) {
    ...
}

왜 프론트는 /app/chat.level1.ping인데 백엔드는 /chat.level1.ping이냐면, Spring이 /app prefix를 제거한 뒤 @MessageMapping을 찾기 때문이다.

프론트 publish destination Spring 처리 백엔드 매핑
/app/chat.level1.ping /app 제거 @MessageMapping("/chat.level1.ping")

4. 백엔드 메서드가 실행됨

메시지 body는 ChatLevel1PingRequest로 변환되어 메서드 파라미터로 들어온다. 

public ChatLevel1PongResponse ping(ChatLevel1PingRequest request) {
    String sender = request.sender() == null || request.sender().isBlank()
            ? "anonymous"
            : request.sender().trim();
    String message = request.message() == null || request.message().isBlank()
            ? "ping"
            : request.message().trim();

    return new ChatLevel1PongResponse(sender, "pong: " + message, Instant.now());
}

이 메서드는 받은 sender/message를 정리한 뒤 ChatLevel1PongResponse를 반환한다.

5. 반환값을 topic으로 전송

메서드 위에 붙은 @SendTo가 반환값을 어디로 보낼지 정한다. 

@SendTo("/topic/chat.level1.pong")

이 뜻은 “메서드 반환값을 /topic/chat.level1.pong으로 보내라”이다.

이 topic은 백엔드 설정에서 simple broker가 관리하도록 등록되어 있다. 

registry.enableSimpleBroker("/topic", "/queue");

그래서 Spring simple broker가 /topic/chat.level1.pong 구독자들에게 메시지를 전달할 수 있다.

6. 프론트 구독자가 응답을 받음

프론트는 이미 /topic/chat.level1.pong을 구독하고 있으므로, 서버가 이 topic으로 메시지를 보내면 subscribe callback이 실행된다. 

client.subscribe("/topic/chat.level1.pong", (frame: IMessage) => {
  const body = JSON.parse(frame.body) as PongMessage;
  appendLog("received", `${body.sender}: ${body.message}`);
});

frame.body에는 백엔드의 ChatLevel1PongResponse가 JSON 문자열로 들어 있다. 그래서 JSON.parse로 객체로 바꿔 사용한다.

주소 매칭 정리

단계 주소 관련 코드 의미
프론트 발행 /app/chat.level1.ping client.publish(...) 서버 로직을 호출하기 위해 메시지를 보냄
백엔드 수신 /chat.level1.ping @MessageMapping /app prefix 제거 후 메서드 매칭
백엔드 응답 /topic/chat.level1.pong @SendTo 반환값을 topic으로 전송
프론트 구독 /topic/chat.level1.pong client.subscribe(...) topic 메시지를 받는 callback 등록

REST 요청-응답과 다른 점

구분 REST STOMP WebSocket
요청 POST /api/... publish("/app/...")
서버 처리 @PostMapping @MessageMapping
응답 HTTP response body @SendTo("/topic/...")로 broker에 발행
클라이언트 수신 await api.post(...) 결과 subscribe(..., callback) 실행
STOMP에서는 보낸 함수가 직접 반환값을 받는 느낌이 아니라, 미리 구독해 둔 topic으로 서버 응답이 흘러오고 callback이 실행되는 구조다.

실제 Level 1에서 확인할 것

  1. 프론트가 /topic/chat.level1.pong을 구독한다.
  2. 프론트가 /app/chat.level1.ping으로 메시지를 보낸다.
  3. 백엔드 @MessageMapping 메서드가 실행된다.
  4. 백엔드가 ChatLevel1PongResponse를 반환한다.
  5. @SendTo가 반환값을 /topic/chat.level1.pong으로 보낸다.
  6. 프론트 subscribe callback이 실행된다.
  7. 이벤트 로그에 received 메시지가 찍힌다.

정리

  • 이 흐름은 구독 및 메시지 응답의 기본 프로세스다.
  • publish("/app/...")는 서버 로직을 호출한다.
  • @MessageMapping은 프론트가 보낸 메시지를 받는다.
  • @SendTo("/topic/...")는 메서드 반환값을 topic으로 보낸다.
  • subscribe("/topic/...")는 topic 메시지를 받을 callback을 등록한다.
  • 받는 주소와 보내는 주소가 다르다는 점이 핵심이다.
