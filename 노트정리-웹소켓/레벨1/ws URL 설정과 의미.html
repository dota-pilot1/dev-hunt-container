WebSocket URL

ws URL 설정과 의미

이 문서는 프론트의 brokerURL: wsUrl과 백엔드의 registry.addEndpoint("/ws")가 어떻게 연결되는지, 그리고 하나의 ws URL로 프로젝트의 WebSocket 기능을 어떻게 관리하는지 정리한다.

핵심 결론

ws://localhost:4101/ws는 WebSocket 연결을 여는 입구다. 이 URL 하나로 서버와 브라우저 사이에 STOMP 연결을 만든다.

하지만 모든 기능이 URL 하나로만 구분되는 것은 아니다. 연결은 /ws 하나로 열고, 그 연결 안에서 /app, /topic, /queue, /user 같은 STOMP destination으로 기능을 나눈다.

REST에서는 URL이 기능을 직접 나누는 경우가 많지만, STOMP WebSocket에서는 연결 URL과 메시지 destination을 분리해서 생각해야 한다.

백엔드 설정

파일:

dev-hunt-server/src/main/java/com/cj/devhunt/chat/config/WebSocketConfig.java

@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("http://localhost:*", "http://127.0.0.1:*");
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.setApplicationDestinationPrefixes("/app");
        registry.enableSimpleBroker("/topic", "/queue");
        registry.setUserDestinationPrefix("/user");
    }
}

프론트 설정

파일:

dev-hunt-frontend/src/features/chat-study-level1/ChatStudyLevel1Page.tsx

function getWebSocketUrl() {
  const apiUrl = process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:4101";
  return `${apiUrl.replace(/^http/, "ws").replace(/\/$/, "")}/ws`;
}

const wsUrl = useMemo(getWebSocketUrl, []);

const client = new Client({
  brokerURL: wsUrl,
  ...
});

client.activate();

NEXT_PUBLIC_API_URL이 없으면 기본 API 서버 주소 http://localhost:4101을 사용하고, 이를 WebSocket 프로토콜인 ws://localhost:4101로 바꾼 뒤 /ws를 붙인다.

서로 어떻게 매칭되나?

개념 백엔드 프론트 의미
WebSocket 접속 입구 registry.addEndpoint("/ws") brokerURL: "ws://localhost:4101/ws" 브라우저가 서버와 WebSocket 연결을 여는 주소
서버 메서드 호출 prefix setApplicationDestinationPrefixes("/app") publish({ destination: "/app/chat.level1.ping" }) 클라이언트 메시지를 @MessageMapping으로 보내는 prefix
브로드캐스트 구독 prefix enableSimpleBroker("/topic") subscribe("/topic/chat.level1.pong") 여러 클라이언트가 같은 메시지를 받을 수 있는 topic
큐 prefix enableSimpleBroker("/queue") Level 1 미사용 보통 1:1 또는 큐 성격 메시지에 사용
사용자별 prefix setUserDestinationPrefix("/user") Level 1 미사용 로그인 사용자별 개인 메시지로 확장할 때 사용

/ws는 무엇인가?

/ws는 STOMP 메시지를 보내는 destination이 아니라, WebSocket 연결을 처음 여는 handshake endpoint다.

브라우저는 이 주소로 접속해서 서버와 긴 연결을 만든다. 연결이 열린 뒤에는 이 연결 안에서 STOMP 프레임이 오간다. 

브라우저
  -- WebSocket handshake -->
서버 /ws endpoint

연결 성공 후

브라우저
  -- STOMP SEND / SUBSCRIBE / DISCONNECT frame -->
서버

프로젝트 WebSocket 로직을 ws URL 하나로 모두 관리하나?

연결 입구는 하나로 관리할 수 있다. 현재 프로젝트에서는 모든 STOMP WebSocket 연결이 /ws endpoint로 들어오게 만들었다.

하지만 기능 구분은 /ws URL이 아니라 STOMP destination으로 한다.

구분 예시 역할
연결 URL ws://localhost:4101/ws 서버와 브라우저 사이의 WebSocket 연결 생성
서버 호출 destination /app/chat.level1.ping 서버의 @MessageMapping 메서드 실행
구독 destination /topic/chat.level1.pong 서버가 보낸 메시지를 프론트가 받음
따라서 “ws URL 하나로 전부 관리한다”는 말은 반만 맞다. 연결 endpoint는 하나로 관리하고, 실제 채팅 기능들은 STOMP destination으로 나눠 관리한다.

비유로 이해하기

WebSocket/STOMP 구조를 전화 연결에 비유하면 다음과 같다.

개념 비유
/ws 전화선을 연결하는 대표 번호
/app/... 상담원에게 특정 업무를 요청하는 내선 번호
/topic/... 공지 방송 채널
/queue/... 개별 대기열 또는 1:1 전달 채널
/user/... 특정 사용자 전용 개인 채널

Level 1에서의 실제 흐름

  1. 프론트가 getWebSocketUrl()로 ws://localhost:4101/ws를 만든다.
  2. new Client({ brokerURL: wsUrl })로 STOMP client를 만든다.
  3. client.activate()로 /ws endpoint에 연결한다.
  4. 연결 성공 후 client.subscribe("/topic/chat.level1.pong", callback)을 실행한다.
  5. Ping 버튼 클릭 시 client.publish({ destination: "/app/chat.level1.ping" })을 실행한다.
  6. 백엔드 @MessageMapping("/chat.level1.ping")이 메시지를 처리한다.
  7. 백엔드 @SendTo("/topic/chat.level1.pong")이 응답을 topic으로 보낸다.
  8. 프론트의 subscribe callback이 응답을 받아 로그에 출력한다.

왜 /ws 하나로 두는가?

STOMP는 WebSocket 연결 하나 위에서 여러 destination을 주고받을 수 있다. 그래서 기능마다 /ws-chat, /ws-alarm, /ws-video처럼 endpoint를 나누지 않아도 된다.

일반적으로는 endpoint를 하나로 두고 destination을 기능별로 나누는 쪽이 관리하기 쉽다. 

연결 endpoint
  /ws

기능별 destination
  /app/chat.level1.ping
  /app/chat.rooms.send
  /topic/chat.rooms.{roomId}
  /user/queue/notifications

나중에 기능이 늘어나면?

연결 URL은 그대로 /ws를 쓰고, destination만 늘리는 방식으로 확장할 수 있다.

기능 프론트 publish 프론트 subscribe
Level 1 ping/pong /app/chat.level1.ping /topic/chat.level1.pong
채팅방 메시지 /app/chat.rooms.send /topic/chat.rooms.{roomId}
입장/퇴장 이벤트 /app/chat.rooms.enter /topic/chat.rooms.{roomId}.events
개인 알림 /app/notifications.read /user/queue/notifications

언제 endpoint를 여러 개로 나누나?

대부분의 채팅 기능은 /ws 하나로 충분하다. endpoint를 나누는 경우는 보통 다음처럼 성격이 크게 다를 때다.

  • 인증 정책이 완전히 다른 WebSocket 연결이 필요할 때
  • 운영/관리용 WebSocket과 사용자용 WebSocket을 물리적으로 분리하고 싶을 때
  • 서버 인프라, 라우팅, 모니터링 정책을 endpoint별로 다르게 가져가야 할 때
학습 단계와 일반적인 서비스 채팅에서는 endpoint를 여러 개 만들기보다 /ws 하나로 시작하는 편이 단순하고 좋다.

정리

  • /ws는 WebSocket 연결을 여는 입구다.
  • 프론트의 brokerURL: wsUrl은 백엔드의 addEndpoint("/ws")와 매칭된다.
  • /app, /topic, /queue, /user는 연결 URL이 아니라 STOMP destination prefix다.
  • 프로젝트의 WebSocket 연결 endpoint는 하나로 둘 수 있다.
  • 실제 기능 구분은 destination으로 한다.
  • Level 1에서는 /ws, /app/chat.level1.ping, /topic/chat.level1.pong 세 개를 이해하면 된다.
