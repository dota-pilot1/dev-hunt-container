Level 2

보내기 버튼 관련 로직 정리

이 문서는 ChatStudyLevel2Page.tsx에서 사용자가 메시지를 입력하고 보내기 버튼을 눌렀을 때, 프론트 상태 검사부터 STOMP publish, 백엔드 message mapping, topic broadcast, 클라이언트 렌더링까지 이어지는 흐름을 정리한다.

한 줄 요약

보내기 버튼은 직접 서버 API를 호출하는 버튼이 아니라, 이미 연결된 STOMP client로 /app/chat.level2.send에 메시지를 발행하는 트리거다. 서버는 이 메시지를 받아 /topic/chat.level2.messages로 다시 뿌리고, 프론트는 구독 콜백에서 받은 메시지를 messages state에 추가해서 화면에 표시한다.

큰 그림 먼저 보기

[브라우저 화면]
  입력창 draft
  보내기 버튼
      │
      │ submit event
      ▼
[sendMessage 함수]
  기본 submit 새로고침 막기
  clientRef.current 확인
  draft.trim() 확인
  STOMP publish
      │
      │ destination: /app/chat.level2.send
      ▼
[Spring WebSocket Controller]
  @MessageMapping("/chat.level2.send")
  ChatLevel2MessageRequest 수신
  ChatLevel2MessageResponse 생성
      │
      │ @SendTo("/topic/chat.level2.messages")
      ▼
[구독 중인 모든 브라우저]
  client.subscribe 콜백 실행
  frame.body JSON.parse
  setMessages(...)
      │
      ▼
[채팅 말풍선 렌더링]

여기서 포인트는 보내기 버튼 클릭과 화면에 메시지가 추가되는 시점이 같은 함수 안에 있지 않다는 점이다. 버튼은 publish만 하고, 화면 추가는 서버가 다시 내려준 topic 메시지를 받은 subscribe 콜백에서 일어난다.

사용자가 보는 화면 기준 흐름

1. 연결

연결 버튼을 눌러 STOMP client가 활성화되고 topic 구독이 잡힌다.

2. 입력

입력창 값이 draft state에 저장된다.

3. 보내기

보내기 버튼 또는 form submit이 sendMessage()를 실행한다.

4. 발행

프론트가 /app/chat.level2.send로 JSON payload를 publish한다.

5. 브로드캐스트

서버가 메시지 DTO를 만들고 /topic/chat.level2.messages로 전송한다.

6. 렌더링

구독 중인 모든 클라이언트가 메시지를 받아 채팅 목록에 추가한다.

이벤트 흐름을 코드 기준으로 쪼개기

레벨2에서 실제로 중요한 이벤트는 세 가지다. 입력 이벤트, submit 이벤트, 구독 메시지 수신 이벤트다.

이벤트 발생 시점 실행되는 코드 결과
입력 이벤트 사용자가 input에 글자를 입력할 때 onChange={(event) => setDraft(event.target.value)} draft state가 바뀌고, 버튼 활성화 조건도 같이 갱신된다.
submit 이벤트 보내기 버튼 클릭 또는 Enter 입력 <form onSubmit={sendMessage}> sendMessage()가 실행되어 STOMP publish를 시도한다.
수신 이벤트 서버가 topic으로 메시지를 브로드캐스트할 때 client.subscribe(..., callback) messages state에 새 메시지가 추가되고 화면이 다시 렌더링된다.
React 입장에서 보면 draft와 messages는 화면을 다시 그리게 만드는 state다. 반대로 clientRef와 subscriptionRef는 STOMP 객체를 보관하는 저장소라서, 값이 바뀌어도 그 자체로 화면 렌더링을 유발하지 않는다.

보내기 버튼 JSX

버튼은 <form onSubmit={sendMessage}> 안에 있다. 그래서 버튼 클릭뿐 아니라 입력창에서 Enter를 눌러도 같은 sendMessage() 함수가 실행된다. 

<form onSubmit={sendMessage}>
  <input
    value={draft}
    onChange={(event) => setDraft(event.target.value)}
    disabled={!connected}
    placeholder={connected ? "메시지를 입력하세요" : "먼저 연결하세요"}
  />

  <button
    type="submit"
    disabled={!connected || !draft.trim()}
  >
    <Send />
    보내기
  </button>
</form>

버튼이 비활성화되는 조건

조건 의미 이유
!connected STOMP 연결이 아직 완료되지 않음 연결 전에는 client.publish()를 해도 보낼 수 없으므로 버튼을 막는다.
!draft.trim() 입력값이 비어 있거나 공백뿐임 의미 없는 빈 메시지가 서버로 가지 않게 프론트에서 먼저 차단한다.
중요: 버튼의 disabled는 사용자 경험을 위한 1차 방어다. 실제 함수 안에서도 client?.connected와 content를 다시 검사한다. UI 상태와 실제 STOMP client 상태가 순간적으로 어긋날 수 있기 때문이다.

sendMessage 함수 전체 흐름

const sendMessage = (event?: FormEvent<HTMLFormElement>) => {
  event?.preventDefault();

  const client = clientRef.current;
  const content = draft.trim();
  if (!client?.connected || !content) return;

  client.publish({
    destination: "/app/chat.level2.send",
    body: JSON.stringify({ sender, content }),
  });
  setDraft("");
};

라인별 설명

코드 역할
event?.preventDefault() form submit의 기본 동작인 페이지 새로고침을 막는다. 채팅 화면은 새로고침 없이 STOMP로만 메시지를 보내야 한다.
const client = clientRef.current 연결 때 만들어둔 STOMP client 객체를 ref에서 꺼낸다. ref를 쓰는 이유는 렌더링과 무관하게 같은 client 인스턴스를 유지하기 위해서다.
const content = draft.trim() 입력값 앞뒤 공백을 제거한다. 공백뿐인 메시지는 빈 문자열이 된다.
if (!client?.connected || !content) return STOMP 연결이 없거나 메시지가 비어 있으면 발행하지 않고 종료한다.
client.publish(...) 서버의 application destination으로 메시지를 발행한다. 여기서 실제 WebSocket 송신이 발생한다.
setDraft("") 전송 후 입력창을 비운다. 화면에 메시지를 즉시 추가하지는 않는다.
왜 바로 messages에 추가하지 않을까? 이 구현에서는 내가 보낸 메시지도 서버가 topic으로 다시 브로드캐스트한 뒤 구독 콜백에서 추가한다. 이렇게 하면 내 화면과 다른 사람 화면이 같은 응답 DTO를 기준으로 렌더링된다. 즉, id와 sentAt은 클라이언트가 임의로 만드는 값이 아니라 서버가 확정한 값이다.

클릭 후 실제 실행 순서

사용자가 보내기를 누르면 아래 순서로 실행된다. 브라우저 화면에서 보면 한 번에 일어나는 것처럼 보이지만, 실제로는 프론트 송신과 서버 응답 수신이 분리되어 있다.

  1. 버튼의 type="submit" 때문에 form submit 이벤트가 발생한다.
  2. React가 onSubmit={sendMessage}에 연결된 sendMessage 함수를 호출한다.
  3. event.preventDefault()로 브라우저의 기본 form 제출, 즉 페이지 새로고침을 막는다.
  4. clientRef.current에서 현재 STOMP client를 꺼낸다.
  5. draft.trim()으로 입력값을 정리한다.
  6. 연결이 끊겨 있거나 내용이 비어 있으면 return으로 종료한다.
  7. 정상 조건이면 client.publish()로 서버에 메시지를 보낸다.
  8. 보낸 직후 setDraft("")로 입력창만 비운다.
  9. 서버가 메시지를 처리하고 topic으로 다시 전송한다.
  10. 프론트의 subscribe 콜백이 실행되고, 그때 setMessages()로 화면 목록에 추가된다.

state와 ref 변화

이름 종류 언제 바뀌나 화면 영향
draft state 입력창에 글자를 입력할 때, 전송 후 setDraft("") 할 때 입력창 값과 보내기 버튼 활성화 여부가 바뀐다.
messages state subscribe 콜백에서 서버 메시지를 받을 때 채팅 말풍선 목록이 다시 렌더링된다.
connectionState state 연결 시도, 연결 성공, 연결 해제, 에러 발생 시 상단 연결 배지, 버튼 disabled, 입력창 disabled가 바뀐다.
clientRef ref connect()에서 STOMP client를 만들 때 저장된다. 직접 렌더링을 일으키지는 않는다. sendMessage()가 publish할 client를 찾는 데 쓴다.
subscriptionRef ref client.subscribe()의 반환값을 저장할 때 직접 렌더링을 일으키지는 않는다. 연결 해제나 cleanup 때 구독을 끊는 데 쓴다.

publish payload 구조

보내기 버튼이 눌리면 프론트는 아래 JSON을 문자열로 바꿔 STOMP frame body에 넣는다. 

{
  "sender": "오현석",
  "content": "안녕하세요"
}
필드 값을 만드는 곳 설명
sender user?.username ?? user?.email ?? "tester" 로그인 사용자 이름을 우선 사용하고, 없으면 이메일, 그것도 없으면 테스트용 이름을 사용한다.
content draft.trim() 입력창의 현재 값을 정리한 메시지 본문이다.

STOMP 경로가 바뀌는 방식

프론트 코드에서는 destination: "/app/chat.level2.send"로 보낸다. 하지만 백엔드 컨트롤러에는 @MessageMapping("/chat.level2.send")만 적혀 있다. 

// 프론트
client.publish({
  destination: "/app/chat.level2.send",
  body: JSON.stringify({ sender, content }),
});

// 백엔드
@MessageMapping("/chat.level2.send")
@SendTo("/topic/chat.level2.messages")
public ChatLevel2MessageResponse sendMessage(ChatLevel2MessageRequest request) {
  ...
}
/app은 WebSocketConfig에서 설정한 application destination prefix다. 그래서 프론트는 /app/chat.level2.send로 보내지만, Spring은 prefix를 떼고 /chat.level2.send에 매핑된 메서드를 찾는다.

백엔드 처리 흐름

@MessageMapping("/chat.level2.send")
@SendTo("/topic/chat.level2.messages")
public ChatLevel2MessageResponse sendMessage(ChatLevel2MessageRequest request) {
    String sender = request.sender() == null || request.sender().isBlank()
            ? "anonymous"
            : request.sender().trim();
    String content = request.content() == null || request.content().isBlank()
            ? "(empty)"
            : request.content().trim();

    return new ChatLevel2MessageResponse(
            UUID.randomUUID().toString(),
            sender,
            content,
            Instant.now()
    );
}
  1. STOMP broker가 /app/chat.level2.send 메시지를 application handler로 넘긴다.
  2. Spring이 JSON body를 ChatLevel2MessageRequest record로 변환한다.
  3. 컨트롤러가 sender와 content를 정리한다.
  4. 서버가 메시지 id와 sentAt을 새로 만든다.
  5. @SendTo에 의해 반환값이 /topic/chat.level2.messages로 전송된다.

구독 콜백과 화면 반영

메시지를 화면에 추가하는 곳은 보내기 버튼 함수가 아니라, 연결 성공 후 등록한 subscribe 콜백이다. 

subscriptionRef.current = client.subscribe(
  "/topic/chat.level2.messages",
  (frame: IMessage) => {
    const body = JSON.parse(frame.body) as ChatMessage;
    setMessages((prev) => [...prev, body].slice(-100));
  }
);
코드 설명
client.subscribe(...) /topic/chat.level2.messages를 구독한다. 이 topic으로 오는 메시지는 모두 콜백으로 들어온다.
JSON.parse(frame.body) STOMP frame body는 문자열이므로 화면에서 쓰기 위해 객체로 변환한다.
setMessages((prev) => [...prev, body].slice(-100)) 기존 메시지 뒤에 새 메시지를 붙이고, 최근 100개만 유지한다.
결론: 보내기 버튼은 messages를 직접 바꾸지 않는다. 버튼은 서버로 메시지를 보내는 역할만 하고, 화면 반영은 서버가 다시 내려준 topic 메시지를 받은 subscribe 콜백이 담당한다.

내 메시지와 남의 메시지 구분

렌더링할 때는 메시지의 sender와 현재 화면의 sender를 비교한다. 같으면 오른쪽 검은 말풍선, 다르면 왼쪽 일반 말풍선으로 보여준다. 

messages.map((message) => {
  const mine = message.sender === sender;

  return (
    <div className={`flex ${mine ? "justify-end" : "justify-start"}`}>
      ...
    </div>
  );
})
현재는 학습용 구현이라 sender 문자열 비교로 내 메시지를 판단한다. 실전에서는 사용자 ID처럼 변하지 않는 고유 식별자를 응답 DTO에 포함시키는 편이 더 안전하다.

증상 확인할 것 관련 코드
보내기 버튼이 비활성화됨 연결 상태가 connected인지, 입력값이 공백뿐이 아닌지 확인한다. disabled={!connected || !draft.trim()}
버튼을 눌러도 새로고침됨 event.preventDefault()가 실행되는지 확인한다. event?.preventDefault()
서버가 메시지를 못 받음 publish destination이 /app/chat.level2.send인지 확인한다. client.publish({ destination: ... })
서버는 받는데 화면에 안 뜸 subscribe destination이 /topic/chat.level2.messages인지 확인한다. client.subscribe(...)
내 메시지가 왼쪽에 뜸 응답의 sender와 현재 클라이언트의 sender 문자열이 같은지 확인한다. message.sender === sender
