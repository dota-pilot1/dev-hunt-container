Level 2

구독 콜백 메시지 렌더링 로직

이 문서는 서버가 /topic/chat.level2.messages로 메시지를 브로드캐스트했을 때, 프론트가 그 메시지를 받아 채팅 목록에 추가하고 화면에 렌더링하는 부분만 따로 설명한다.

정확한 이름

이 부분은 보통 subscribe callback, 구독 콜백, 또는 메시지 수신 핸들러라고 부르면 된다.

더 풀어서 말하면 STOMP topic 구독 콜백에서 수신 메시지를 state에 반영하는 로직이다.

전체 흐름

connect()
  ↓
client.subscribe("/topic/chat.level2.messages", callback)
  ↓
서버가 topic으로 메시지 broadcast
  ↓
callback(frame) 실행
  ↓
JSON.parse(frame.body)
  ↓
setMessages((prev) => [...prev, body].slice(-100))
  ↓
React 재렌더링
  ↓
messages.map(...)으로 말풍선 표시
핵심: 보내기 버튼은 /app/chat.level2.send로 메시지를 보내는 역할이다. 화면에 메시지를 추가하는 역할은 보내기 버튼이 아니라, /topic/chat.level2.messages를 구독해둔 콜백이 담당한다.

구독을 등록하는 코드

구독 등록은 연결 성공 시점인 onConnect 안에서 한다. 연결이 되기 전에는 topic을 구독할 수 없기 때문이다. 

onConnect: () => {
  setConnectionState("connected");
  setSystemMessage("연결되었습니다. 다른 브라우저 탭에서도 같은 페이지를 열어 브로드캐스트를 확인할 수 있습니다.");

  subscriptionRef.current = client.subscribe(
    "/topic/chat.level2.messages",
    (frame: IMessage) => {
      const body = JSON.parse(frame.body) as ChatMessage;
      setMessages((prev) => [...prev, body].slice(-100));
    }
  );
}

콜백 내부 로직

코드 역할
client.subscribe(...) 서버 topic을 계속 듣겠다고 등록한다. 이 등록이 끝나면 해당 topic으로 오는 메시지마다 콜백이 실행된다.
"/topic/chat.level2.messages" 서버가 @SendTo로 메시지를 뿌리는 주소다. 여러 클라이언트가 같은 topic을 구독할 수 있다.
frame STOMP가 넘겨주는 수신 메시지 객체다. 실제 JSON 문자열은 frame.body 안에 들어 있다.
JSON.parse(frame.body) 문자열로 온 JSON을 프론트에서 쓸 수 있는 객체로 바꾼다.
setMessages((prev) => ...) 기존 메시지 목록을 기준으로 새 메시지를 뒤에 붙인다. 함수형 업데이트를 쓰면 이전 state를 안전하게 참조할 수 있다.
.slice(-100) 메시지 목록이 무한히 커지지 않도록 최근 100개만 유지한다.

입력창에 추가하는 게 아니라 messages 목록에 추가한다

표현을 정확히 하면, 서버에서 온 메시지는 입력창에 추가되는 것이 아니다. 입력창은 draft state가 담당한다. 서버에서 온 메시지는 messages state에 추가되고, 그 messages를 화면에서 반복 렌더링해서 채팅 말풍선으로 보여준다.

화면 영역 state 역할
하단 입력창 draft 사용자가 아직 보내기 전 입력 중인 텍스트
채팅 메시지 목록 messages 서버에서 수신 완료된 메시지들
따라서 “구독했던 연결망으로 메시지가 날아오면 입력창에 추가된다”보다는 “구독한 topic으로 메시지가 오면 수신 콜백이 실행되고, 그 메시지를 messages state에 추가해서 채팅 목록에 렌더링한다”가 정확한 표현이다.

서버 응답 body 형태

콜백에서 JSON.parse(frame.body) 하면 아래 형태의 객체가 된다. 

{
  "id": "cc7ae965-3e5c-4f40-b7db-08af9a5f77d6",
  "sender": "오현석",
  "content": "안녕하세요",
  "sentAt": "2026-06-27T10:52:28.741791Z"
}

왜 setMessages는 함수형 업데이트를 쓰나

setMessages((prev) => [...prev, body].slice(-100));

WebSocket 메시지는 사용자의 클릭과 별개로 비동기적으로 들어온다. 그래서 현재 렌더 시점의 messages 값을 직접 참조하기보다, React가 넘겨주는 가장 최신 이전 값인 prev를 기준으로 새 메시지를 붙이는 편이 안전하다.

화면 렌더링 코드

messages state가 바뀌면 React가 다시 렌더링하고, 아래 messages.map 부분이 새 목록 기준으로 다시 그려진다. 

{messages.map((message) => {
  const mine = message.sender === sender;

  return (
    <div
      key={message.id}
      className={`flex ${mine ? "justify-end" : "justify-start"}`}
    >
      <div>
        <div>
          <span>{message.sender}</span>
          <span>{new Date(message.sentAt).toLocaleTimeString("ko-KR")}</span>
        </div>
        <p>{message.content}</p>
      </div>
    </div>
  );
})}

이 로직을 부를 때 추천 표현

  • 구독 콜백
  • STOMP subscribe 콜백
  • topic 메시지 수신 핸들러
  • 수신 메시지 state 반영 로직
  • WebSocket 메시지 렌더링 로직
제일 정확하게 말하면:
“STOMP topic 구독 콜백에서 서버가 브로드캐스트한 메시지를 받아 messages state에 추가하고, React가 그 state를 기준으로 채팅 목록을 다시 렌더링한다.”
