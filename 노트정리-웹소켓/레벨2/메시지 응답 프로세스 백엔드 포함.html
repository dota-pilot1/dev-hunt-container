Level 2

메시지 응답 프로세스: 백엔드 포함

이 문서는 client.subscribe("/topic/chat.level2.messages", callback)를 중심으로, 프론트가 미리 구독을 걸어두고 백엔드가 메시지를 브로드캐스트하면 그 응답이 어떻게 다시 화면에 반영되는지 정리한다.

질문에 대한 답

맞다. 아래처럼 콜백을 설정해두면 /topic/chat.level2.messages 구독에 대한 메시지가 올 때마다 콜백 함수가 실행되고, 그 안에서 setMessages()가 호출되어 채팅 목록에 메시지가 추가된다. 

subscriptionRef.current = client.subscribe("/topic/chat.level2.messages", (frame: IMessage) => {
  const body = JSON.parse(frame.body) as ChatMessage;
  setMessages((prev) => [...prev, body].slice(-100));
});

응답 프로세스 전체 그림

[프론트 연결 성공]
  onConnect 실행
      ↓
[프론트 구독 등록]
  client.subscribe("/topic/chat.level2.messages", callback)
      ↓
  이제 이 브라우저는 해당 topic을 듣는 상태가 됨

[사용자 메시지 전송]
  client.publish("/app/chat.level2.send", JSON)
      ↓
[백엔드 수신]
  @MessageMapping("/chat.level2.send")
      ↓
  ChatLevel2MessageRequest로 변환
      ↓
  ChatLevel2MessageResponse 생성
      ↓
[백엔드 응답 브로드캐스트]
  @SendTo("/topic/chat.level2.messages")
      ↓
[프론트 구독 콜백 실행]
  callback(frame)
      ↓
  JSON.parse(frame.body)
      ↓
  setMessages(...)
      ↓
[화면 반영]
  messages.map(...)으로 말풍선 렌더링
중요한 점은 subscribe가 서버에 즉시 메시지를 요청하는 코드가 아니라는 것이다. subscribe는 “앞으로 이 topic으로 오는 메시지를 받겠다”는 수신 통로를 열어두는 코드다.

1단계: 프론트가 먼저 구독 통로를 열어둔다

구독은 onConnect 안에서 등록된다. WebSocket/STOMP 연결이 성공한 뒤에야 topic을 구독할 수 있기 때문이다. 

onConnect: () => {
  setConnectionState("connected");
  setSystemMessage("연결되었습니다. 다른 브라우저 탭에서도 같은 페이지를 열어 브로드캐스트를 확인할 수 있습니다.");

  subscriptionRef.current = client.subscribe("/topic/chat.level2.messages", (frame: IMessage) => {
    const body = JSON.parse(frame.body) as ChatMessage;
    setMessages((prev) => [...prev, body].slice(-100));
  });
}
코드 의미
client.subscribe(...) STOMP broker에게 특정 topic을 구독하겠다고 등록한다.
"/topic/chat.level2.messages" 백엔드가 응답을 뿌리는 주소다. 이 주소를 구독한 클라이언트만 메시지를 받는다.
(frame) => { ... } 해당 topic으로 메시지가 도착했을 때 실행될 콜백 함수다.
subscriptionRef.current = ... 나중에 연결 해제할 때 unsubscribe() 할 수 있도록 구독 객체를 보관한다.

2단계: 사용자가 메시지를 보내면 application destination으로 간다

보내기 버튼은 아래 코드로 서버에 메시지를 발행한다. 이때 목적지는 구독 주소인 /topic/...이 아니라 서버 컨트롤러가 받는 /app/... 주소다. 

client.publish({
  destination: "/app/chat.level2.send",
  body: JSON.stringify({ sender, content }),
});
/app/chat.level2.send는 “서버로 보내는 주소”이고, /topic/chat.level2.messages는 “서버가 클라이언트들에게 뿌리는 주소”다. 둘은 역할이 다르다.

3단계: 백엔드가 publish 메시지를 받는다

프론트가 /app/chat.level2.send로 보낸 메시지는 Spring의 application destination prefix인 /app이 제거된 뒤 @MessageMapping("/chat.level2.send") 메서드로 들어온다. 

@MessageMapping("/chat.level2.send")
@SendTo("/topic/chat.level2.messages")
public ChatLevel2MessageResponse sendMessage(ChatLevel2MessageRequest request) {
    String sender = request.sender() == null || request.sender().isBlank()
            ? "anonymous"
            : request.sender().trim();
    String content = request.content() == null || request.content().isBlank()
            ? "(empty)"
            : request.content().trim();

    return new ChatLevel2MessageResponse(
            UUID.randomUUID().toString(),
            sender,
            content,
            Instant.now()
    );
}
백엔드 코드 역할
@MessageMapping("/chat.level2.send") 프론트 publish 메시지를 받는 진입점이다. 프론트 주소는 /app/chat.level2.send다.
ChatLevel2MessageRequest request STOMP frame body의 JSON이 Java record로 변환된 결과다.
UUID.randomUUID().toString() 화면 렌더링 key로 쓸 수 있는 메시지 ID를 서버에서 만든다.
Instant.now() 메시지 전송 시각을 서버 기준으로 만든다.
@SendTo("/topic/chat.level2.messages") 메서드 반환값을 해당 topic으로 브로드캐스트한다.

4단계: 백엔드 응답 DTO가 topic으로 브로드캐스트된다

백엔드는 아래 형태의 응답을 만든다. 이 객체가 JSON으로 직렬화되어 /topic/chat.level2.messages를 구독 중인 클라이언트들에게 전송된다. 

public record ChatLevel2MessageResponse(
        String id,
        String sender,
        String content,
        Instant sentAt
) {
}
{
  "id": "cc7ae965-3e5c-4f40-b7db-08af9a5f77d6",
  "sender": "오현석",
  "content": "안녕하세요",
  "sentAt": "2026-06-27T10:52:28.741791Z"
}

5단계: 프론트 구독 콜백이 응답을 받는다

서버가 topic으로 응답을 뿌리는 순간, 이미 구독을 걸어둔 프론트의 콜백이 실행된다. 

(frame: IMessage) => {
  const body = JSON.parse(frame.body) as ChatMessage;
  setMessages((prev) => [...prev, body].slice(-100));
}
처리 설명
frame STOMP가 넘겨준 수신 frame이다. headers와 body를 가진다.
frame.body 서버 응답 JSON 문자열이다.
JSON.parse(frame.body) 문자열을 ChatMessage 객체로 바꾼다.
setMessages((prev) => [...prev, body].slice(-100)) 기존 메시지 목록 뒤에 새 메시지를 추가하고 최근 100개만 유지한다.
이 콜백이 실행되는 시점이 “서버 응답이 프론트에 도착한 시점”이다. 사용자가 보낸 메시지도 이 콜백을 통해 화면에 추가된다.

6단계: React가 messages 변경을 감지하고 다시 그린다

setMessages()가 호출되면 React state가 바뀐다. 그러면 컴포넌트가 다시 렌더링되고, messages.map(...)이 새 메시지 목록을 기준으로 말풍선을 만든다. 

{messages.map((message) => {
  const mine = message.sender === sender;

  return (
    <div key={message.id} className={`flex ${mine ? "justify-end" : "justify-start"}`}>
      ...
      <p>{message.content}</p>
    </div>
  );
})}

요청 주소와 응답 주소 비교

구분 주소 방향 담당 코드
메시지 요청 /app/chat.level2.send 프론트 → 백엔드 client.publish(), @MessageMapping
메시지 응답 /topic/chat.level2.messages 백엔드 → 구독 중인 프론트들 @SendTo, client.subscribe()

이 프로세스를 부르는 말

  • STOMP publish/subscribe 응답 흐름
  • topic broadcast 수신 흐름
  • 구독 콜백 기반 메시지 렌더링 흐름
  • WebSocket 메시지 응답 처리 프로세스
제일 정확한 설명:
프론트는 /topic/chat.level2.messages를 미리 구독해두고, 백엔드는 /app/chat.level2.send로 받은 메시지를 처리한 뒤 @SendTo로 같은 topic에 브로드캐스트한다. 그러면 구독 콜백이 실행되어 messages state에 응답 메시지를 추가한다.
